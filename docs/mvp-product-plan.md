# AI Sticker Maker MVP 产品方案

## 目标

第一版只验证一个核心能力：

> 用户选择数量，输入主题或抽方向，系统自动生成表情条目，并生成一张带中文文字的表情包网格大图。

MVP 先不考虑微信上架、积分、切图、单张补图和投稿包导出。

## 目标用户

第一版先面向创作者，不主打普通个人照片转表情。

主要用户：

- 微信表情投稿创作者
- 小红书、视频号、公众号等轻 IP 创作者
- 想给账号做统一角色表情包的运营者

## 产品形态

第一版先做网页 Web App，不先做微信小程序。

原因：

- 图片预览和文件下载更适合网页。
- MVP 阶段需要快速调整流程。
- 当前重点是验证含字网格大图生成效果，不是上架闭环。

## MVP 核心流程

### 1. 选择数量

用户在最开始选择表情数量：

| 数量 | 网格 |
|---:|---|
| 8 | 2 x 4 |
| 16 | 4 x 4 |
| 24 | 4 x 6 |

数量必须前置，因为它决定：

- 需要多少个表情条目
- 网格规格
- 大图 prompt 的结构
- 最终下载结果的布局

### 2. 输入主题或抽方向

用户有想法时，直接输入主题，例如：

```text
一只嘴贱的加班橘猫，适合微信聊天
```

用户也可以抽方向卡。主题是可选约束：

- 如果用户输入了主题，方向卡围绕当前主题发散。
- 如果用户没有输入主题，方向卡随机给出创作方向。
- 用户选中方向卡后，系统会把主题替换成更明确的方向，并继续生成条目。

方向卡只生成低成本结构，不出图。每张方向卡包含：

- 主角
- 性格
- 使用场景
- 画风倾向
- 表情套组方向

方向卡支持刷新。刷新只更换创作方向，不生成图片，也不影响已选数量。

### 3. 自动生成表情条目

系统根据数量、主题或方向卡，自动生成 N 个表情条目，N = 8 / 16 / 24。

用户不需要自己写满表情条目。

每个条目在系统内部包含：

- 画面文字
- 动作描述

用户界面只默认暴露紧凑的“画面文字”输入。动作描述由系统根据主题和角色自动生成，不在 MVP 页面展示，避免用户被大量条目压住。

规则：

- 如果用户没写条目，系统自由生成完整条目。
- 如果用户只改了少数画面文字，系统保留用户修改。
- 用户不需要手工填写动作描述。

### 4. 生成含字网格大图

用户确认条目后，系统调用生图模型生成一张网格大图。

大图要求：

- 所有格子使用同一原创角色和统一画风。
- 每格是一个独立表情。
- 每格包含对应中文文字。
- 中文文字由生图模型直接画入图中，不做系统后期叠字。
- 角色和文字尽量不要跨格。
- 背景简单，适合后续人工切图。

### 5. 下载大图

用户下载生成的大图。

MVP 到这里结束。后续用户自己切图、修图、整理和投稿。

## MVP 页面

第一版低保真原型只需要 3 步：

1. **创建页**
   - 选择数量：8 / 16 / 24
   - 输入主题
   - 抽方向卡
   - 刷新方向卡

2. **条目页**
   - 展示系统自动生成的表情条目
   - 用户主要编辑每格画面文字
   - 条目以多列紧凑网格展示，避免 24 张变成 24 行长表单
   - 点击生成含字大图

3. **大图页**
   - 展示含字网格大图
   - 下载大图

## MVP 不做

- 积分系统
- 支付
- 微信上架
- 微信账号授权
- 投稿包导出
- 自动切图
- 切图状态标记
- 单张补图
- 角色候选
- 角色卡
- 图片存储管理
- 高级审核
- 动态 GIF
- 上传参考图
- 系统后期叠字
- 自部署生图模型

## 外部能力

MVP 实现时至少需要：

- 文本模型 API：根据主题和数量生成表情条目。
- 生图模型 API：根据条目生成含字网格大图。
- 文件下载能力：让用户下载生成的大图。

生图 API Key 必须放在服务端环境变量中，不能暴露到前端。

## 成功标准

- 用户能选择 8 / 16 / 24 张。
- 用户不需要手工想满 24 个表情条目。
- 系统能根据主题或方向卡生成完整条目。
- 用户能生成一张含中文文字的网格大图。
- 用户能下载大图。
- 免费/未付费阶段不讨论，MVP 先验证核心生成链路。
