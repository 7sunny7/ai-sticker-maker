# MVP 原型流程审查

审查对象：`prototype/index.html`

审查时间：2026-07-08

审查范围：创建页 -> 条目页 -> 大图页

## 流程步骤

### 1. 创建

健康度：基本清晰，但入口关系还需要再收紧。

当前页面让用户先选择数量，再输入主题或抽方向。这个顺序是对的，因为数量会影响条目数量和网格布局。主题输入和抽方向也已经被解释为两种入口：有主题时围绕主题发散，没主题时随机抽方向。

主要问题：

- “生成表情条目”和“围绕主题抽方向”同时出现时，用户需要判断自己应该直接生成，还是先抽方向。虽然文案解释了规则，但两个按钮仍然是并列关系，容易让用户犹豫。
- 方向卡被放在主题输入下方，这符合流程，但如果方向卡展开后用户又修改主题，当前页面没有提示方向卡是否已过期。
- “主题（可选）”是准确的，但默认预填主题会弱化“随机抽方向”的入口。用户可能不知道清空主题后才是随机抽。

建议：

- 把两个按钮改成更明确的主次关系：主按钮为“生成条目”，次按钮为“先抽方向”。
- 在方向卡区域显示来源状态：`基于当前主题` 或 `随机方向`。
- 如果用户在方向卡展开后修改主题，可以隐藏旧方向卡，或提示“主题已变更，请刷新方向”。

### 2. 条目

健康度：已经接近 MVP 需要的轻量程度。

当前条目页只展示每格画面文字，动作和构图交给系统内部处理。这比之前每条三个输入框更合理，尤其适合 16 / 24 张。

主要问题：

- 条目页的核心任务是“快速检查文字”，但页面仍然使用“系统生成的表情条目”这个标题，略偏内部视角。
- 用户能编辑文字，但页面没有说明“这些文字会被模型直接画进图里，错字需要重新生成”。这个风险对含字大图很关键。
- 24 个输入框即使用多列展示，仍然需要一个更快的批量确认方式，例如“全部看起来可以，继续生成”。

建议：

- 标题改成更用户化：`确认图片里的文字`。
- 在条目区增加一句短提示：`这些字会直接出现在图片里，生成后改字需要重新生成。`
- 保留底部“生成含字大图”按钮，这个位置是对的，能避免用户没看条目就生成。

### 3. 大图

健康度：作为低保真验证足够，但和真实生图风险之间还有差距。

大图页现在展示占位网格，并允许下载 SVG。这能表达 MVP 结果形态：一张含字网格大图。

主要问题：

- 当前占位图过于规则，不能暴露真实生图的主要风险：错字、串格、跨格、漏格、风格不统一。
- 页面写“当前 MVP 到这里结束”是清晰的，但用户可能仍会期待下一步“切图”。如果产品刻意不做切图，需要更明确地把下载后的责任说清楚。
- 下载按钮是结果页主行动，位置合理。

建议：

- 在结果页保留一句边界提示：`下载后自行切图和处理；本版本不检查格子质量。`
- 后续真实原型接生图 API 后，应在结果页至少展示“重新生成整张大图”入口，即使 MVP 暂不做补图。

## 主要 UX 发现

1. 创建页的总体顺序是合理的：数量前置，然后主题/方向，然后生成条目。

2. 当前最大认知负担在“主题”和“抽方向”的关系。它们不是互斥关系，而是“主题可作为抽方向的约束”。页面已经解释了这个规则，但按钮并列仍然容易造成犹豫。

3. 条目页已经从复杂表单降级到轻量文字确认，这是正确方向。MVP 不应该让用户维护动作描述。

4. 生成按钮放在条目后面是正确的。它强制用户至少扫过条目列表，降低没确认文字就生成的概率。

5. 步骤时间线不可点击是合理的。当前 MVP 是顺序流程，不需要让用户随意跳步。

6. 结果页符合 MVP 边界，但需要更明确地提醒“本版本不做切图和质量检查”。

## 可访问性风险

这些风险来自源码和可见页面结构，不是完整测试结论：

- 数量选择使用 button，键盘可操作性基础上是可行的，但缺少 `aria-pressed` 表示当前选中状态。
- 步骤时间线是纯展示状态，没有 `aria-current="step"`，读屏用户不一定知道当前处于哪一步。
- 方向卡是 button，语义可操作，但选中方向后自动进入下一步，可能需要给出更明显的状态反馈。
- 表情文字输入框有 `aria-label`，这一点是好的。
- Toast 是 `aria-live`，但持续时间较短，读屏场景下可能不稳定。

建议最小修正：

- 当前步骤加 `aria-current="step"`。
- 数量按钮加 `aria-pressed="true/false"`。
- 方向卡选择后如果自动跳转，保留明确 toast 或页面标题变化。

## 结论

当前 MVP 流程方向是合理的：先选数量，再确定主题/方向，再确认文字，最后生成一张含字网格大图。

还需要优化的不是流程大方向，而是两个细节：

1. 创建页要更清楚地区分“直接生成条目”和“先抽方向”。
2. 条目页要明确告诉用户：这些字会直接画进图片里，生成后改字需要重新生成。

如果只做一轮最小修改，优先改这两点。
